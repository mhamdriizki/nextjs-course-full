## Step 1 — cart-store.ts (client-only, persist)

File baru: `lib/store/cart-store.ts`

```ts
import { create } from "zustand";
import { persist, createJSONStorage } from "zustand/middleware";

export interface CartItem {
  id: string;
  name: string;
  price: number;
  qty: number;
}

interface CartState {
  items: CartItem[];
  addItem: (item: Omit<CartItem, "qty">) => void;
  removeItem: (id: string) => void;
  updateQty: (id: string, qty: number) => void;
  clearCart: () => void;
}

// Client-only store: tidak ada SSR concern (tidak ada data sensitif per-user),
// jadi module-level create() di sini AMAN — beda dengan user-store.ts.
export const useCartStore = create<CartState>()(
  persist(
    (set) => ({
      items: [],
      addItem: (item) =>
        set((state) => {
          const existing = state.items.find((i) => i.id === item.id);
          if (existing) {
            return {
              items: state.items.map((i) =>
                i.id === item.id ? { ...i, qty: i.qty + 1 } : i
              ),
            };
          }
          return { items: [...state.items, { ...item, qty: 1 }] };
        }),
      removeItem: (id) =>
        set((state) => ({ items: state.items.filter((i) => i.id !== id) })),
      updateQty: (id, qty) =>
        set((state) => ({
          items: state.items.map((i) => (i.id === id ? { ...i, qty } : i)),
        })),
      clearCart: () => set({ items: [] }),
    }),
    {
      name: "gym-cart-storage", // localStorage key
      storage: createJSONStorage(() => localStorage),
    }
  )
);
```

Tekankan ke murid: ini **module-level `create()`** — biasanya haram di Next.js SSR (Slide 2), tapi
di sini aman karena datanya bukan data sensitif per-user, murni hidup di browser.

## Step 2 — user-store.ts (Store Factory, SSR-safe)

File baru: `lib/store/user-store.ts`

```ts
import { createStore } from "zustand/vanilla";

export interface Member {
  name: string;
  tier: "Silver" | "Gold" | "Platinum";
}

export interface UserState {
  member: Member | null;
  setMember: (member: Member | null) => void;
}

// Factory: bikin store BARU tiap dipanggil, bukan singleton di module level.
// Ini yang bikin aman dari data leaking antar-request di SSR.
export function createUserStore(initialMember: Member | null = null) {
  return createStore<UserState>()((set) => ({
    member: initialMember,
    setMember: (member) => set({ member }),
  }));
}

export type UserStore = ReturnType<typeof createUserStore>;
```

Sorot ke murid: `createStore` dari `zustand/vanilla`, **bukan** `create` dari `zustand`. Ini bukan
hook — dia store instance biasa, nanti yang bikinnya jadi hook adalah Provider di step berikut.

## Step 3 — Provider (Context + useRef + selector hook)

File baru: `app/providers/user-store-provider.tsx`

```tsx
"use client"

import { createContext, useContext, useRef, type ReactNode } from "react";
import { useStore } from "zustand";
import { createUserStore, type UserStore, type UserState, type Member } from "@/lib/store/user-store";

const UserStoreContext = createContext<UserStore | null>(null);

interface UserStoreProviderProps {
  children: ReactNode;
  initialMember?: Member | null;
}

export function UserStoreProvider({
  children,
  initialMember = null,
}: UserStoreProviderProps) {
  // useRef: jangan bikin store baru tiap render, cukup sekali per Provider instance.
  const storeRef = useRef<UserStore | null>(null);
  if (!storeRef.current) {
    storeRef.current = createUserStore(initialMember);
  }

  return (
    <UserStoreContext.Provider value={storeRef.current}>
      {children}
    </UserStoreContext.Provider>
  );
}

export function useUserStore<T>(selector: (state: UserState) => T): T {
  const store = useContext(UserStoreContext);
  if (!store) {
    throw new Error("useUserStore harus dipakai di dalam <UserStoreProvider>");
  }
  return useStore(store, selector);
}
```

**Pertanyaan buat murid (Q2 di kuis):** "Kenapa `useRef`, bukan langsung `createUserStore(...)` di
badan komponen?" → Karena badan komponen jalan tiap render; kalau langsung dipanggil di situ, tiap
render bikin store baru dan state-nya ke-reset terus. `useRef` memastikan store dibuat **sekali**
per Provider instance (per request, karena Provider ini di-render ulang tiap request baru di SSR).

Update `app/layout.tsx` — donat bolong lagi, sama seperti Modul 8.2, tapi sekarang store-nya diisi

```tsx
import { UserStoreProvider } from "./providers/user-store-provider";
import { getCurrentMember } from "@/lib/data/member";

export default async function RootLayout({ children }: { children: React.ReactNode }) {
  const currentMember = await getCurrentMember();

  return (
    <html lang="id" suppressHydrationWarning>
      <body>
        <ThemeProvider /* ... */>
          <UserProvider>{/* Context dari Modul 8.2, tetap ada buat perbandingan */}
            <UserStoreProvider initialMember={currentMember}>
              <Navbar/>

              <footer>...</footer>
            </UserStoreProvider>

        </ThemeProvider>
      </body>

```

**Poin krusial:** `getCurrentMember()` di sini di-cache (`"use cache"`), jadi boleh langsung
`await` di top-level `RootLayout` tanpa `<Suspense>` — beda dengan `searchParams`/`cookies()` asli
yang wajib Suspense (pelajaran Modul 8.1). Kalau di real project kalian baca `cookies()` beneran,
`getCurrentMember` itu sendiri yang perlu dibungkus Suspense, bukan `RootLayout`-nya.

## Step 6 — Sengaja bikin Hydration Error, lalu fix

Ini bagian paling seru buat kelas (sesuai janji script: "kita bakal sengajain bikin Hydration
Error"). Tulis dulu versi **salah**:

```tsx
"use client"
// ❌ VERSI BUGGY — jangan disave, cuma buat demo
import { useCartStore } from "@/lib/store/cart-store";

export function CartBadge() {
  const itemCount = useCartStore((state) =>
    state.items.reduce((total, item) => total + item.qty, 0)
  );
  return <span>🛒 {itemCount}</span>;
}
```

Suruh murid tambah item ke cart, refresh halaman → buka DevTools Console → tunjukkan error merah
"Hydration failed because the server rendered text didn't match the client". Jelaskan kenapa: di
server, `localStorage` tidak ada, jadi `items` = `[]` → render "🛒 0". Begitu sampai di browser,
`persist` middleware langsung load isi cart dari `localStorage` (misal isinya 3) → browser mau
render "🛒 3" tapi HTML dari server sudah kadung "🛒 0" → React panik.

**Fix (yang sudah ada di project ini)** — `app/components/CartBadge.tsx`:

```tsx
"use client"
import { useState, useEffect } from "react";
import { useCartStore } from "@/lib/store/cart-store";

export function CartBadge() {
  const [hydrated, setHydrated] = useState(false);
  const itemCount = useCartStore((state) =>
    state.items.reduce((total, item) => total + item.qty, 0)
  );

  useEffect(() => {
    setHydrated(true);
  }, []);

  if (!hydrated) {
    return <span>🛒 …</span>; // sama persis di server & client-awal
  }

  return <span>🛒 {itemCount}</span>;
}
```

Trik standarnya: **tunda** render angka asli sampai `useEffect` jalan (yang cuma jalan di
browser, setelah mount). Server dan client-awal sama-sama render "🛒 …" → tidak ada mismatch. Baru
setelah itu, React re-render dengan angka sungguhan.

`app/gym-classes/components/CartSummary.tsx` pakai guard yang sama untuk daftar isi cart lengkap
(bukan cuma counter).

## Setelah Live Coding — Quick Check (slide 7)

1. Kenapa Zustand global store (`create()` di module level) bahaya di Next.js SSR? → Store jadi singleton, shared di semua request — data User A bisa bocor ke User B.
2. Fungsi `useRef` di `UserStoreProvider`? → Pastikan store dibuat SEKALI per Provider instance, bukan tiap render.
3. Kenapa `persist` bisa menyebabkan hydration mismatch? → `localStorage` tidak ada di server, SSR render kosong, lalu client load dari storage → mismatch dengan HTML server.
