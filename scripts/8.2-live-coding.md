# Live Coding — Modul 8.2: React Context di App Router

(Panduan pendamping `8.2.txt`, setelah slide "Masuk ke Sesi Live Coding". Perubahan di bawah
**sudah diterapkan langsung ke project ini** — tinggal jalankan `npm run dev` dan tunjukkan ke
murid, atau ketik ulang dari nol kalau mau full live-typing.)

Target: bikin `UserContext` untuk data member yang login, dipasang di `app/layout.tsx` pakai pola
"Donat Bolong", lalu dikonsumsi oleh komponen client di Navbar.

## Kondisi Awal (tunjukkan ke murid dulu)

Project ini sebenarnya **sudah punya** contoh pola Donat Bolong tanpa disadari murid:
`app/components/ThemeProvider.tsx` (bungkus `next-themes`) dipasang di `app/layout.tsx`, dan
`app/layout.tsx` sendiri **tetap Server Component** — tidak ada `"use client"` di file itu.

Tunjukkan `app/layout.tsx` baris atas: tidak ada `"use client"`. Tanyakan: "Kok bisa `ThemeProvider`
butuh browser API tapi layout-nya sendiri nggak 'kotor' jadi Client Component semua?" — itu clue
buat konsep hari ini.

## Step 1 — Kenapa Context nggak bisa langsung di Server Component

Coba (jangan disave) tulis ini di `app/page.tsx`:

```tsx
import { useContext } from "react";
// export default function Home() {
//   const { member } = useContext(UserContext) // Error!
// }
```

Jelaskan: Server Component cuma fungsi yang jalan sekali di server terus jadi HTML mati — nggak
ada "pohon komponen" hidup di server buat context nyebar. Makanya `useContext` cuma jalan di
Client Component.

## Step 2 — Bikin UserContext (donatnya)

File baru: `app/context/UserContext.tsx`

```tsx
"use client"

import { createContext, useContext, useState, type ReactNode } from "react";

export interface Member {
  name: string;
  tier: "Silver" | "Gold" | "Platinum";
}

interface UserContextValue {
  member: Member | null;
  login: (member: Member) => void;
  logout: () => void;
}

const UserContext = createContext<UserContextValue | null>(null);

export function UserProvider({ children }: { children: ReactNode }) {
  const [member, setMember] = useState<Member | null>({
    name: "Rizki",
    tier: "Gold",
  });

  const login = (member: Member) => setMember(member);
  const logout = () => setMember(null);

  return (
    <UserContext.Provider value={{ member, login, logout }}>
      {children}
    </UserContext.Provider>
  );
}

export function useUser() {
  const ctx = useContext(UserContext);
  if (!ctx) {
    throw new Error("useUser harus dipakai di dalam <UserProvider>");
  }
  return ctx;
}
```

Sorot ke murid: file ini **wajib** `"use client"` di baris paling atas karena pakai `useState` dan
`createContext`. Ini "donat"-nya.

## Step 3 — Pasang di layout.tsx (lubang donatnya)

`app/layout.tsx` — **tidak ditambah `"use client"`**, tetap Server Component:

```tsx
import Navbar from "./components/Navbar"
import './globals.css';
import { Figtree } from "next/font/google";
import { cn } from "@/lib/utils";
import { ThemeProvider } from "./components/ThemeProvider";
import { UserProvider } from "./context/UserContext";

const figtree = Figtree({subsets:['latin'],variable:'--font-sans'});

export default function RootLayout({ children }: { children: React.ReactNode }) {
  return (
    <html lang="id" className={cn("font-sans", figtree.variable)} suppressHydrationWarning>
      <body style={{ margin:0, padding:0, fontFamily: 'Arial, sans-serif' }}>
        <ThemeProvider attribute='class' defaultTheme='system' enableSystem disableTransitionOnChange>
          {/* Donat Bolong: UserProvider ('use client') membungkus children,
              tapi children (Navbar & main) sendiri tetap Server Component
              kecuali yang memang sudah "use client". */}
          <UserProvider>
            <Navbar/>
            <main style={{ padding: '2rem', minHeight: '80vh'}}>
              {children}
            </main>
            <footer style={{textAlign: 'center'}}>&copy; 2026 EasyCoding Next.JS</footer>
          </UserProvider>
        </ThemeProvider>
      </body>
    </html>
  )
}
```

**Poin krusial buat ditekankan:** `children` di sini adalah props yang di-passing dari
`RootLayout` (Server Component) — jadi walaupun secara visual dia "masuk" ke dalam
`<UserProvider>`, `children` **tidak ikut** jadi bagian dari Client Component tree `UserProvider`.
Halaman-halaman seperti `/blog` tetap full Server Component.

## Step 4 — Consumer: UserBadge di Navbar

File baru: `app/components/UserBadge.tsx`

```tsx
"use client"

import { useUser } from "@/app/context/UserContext";

export function UserBadge() {
  const { member, login, logout } = useUser();

  if (!member) {
    return (
      <button onClick={() => login({ name: "Rizki", tier: "Gold" })}>
        Login
      </button>
    );
  }

  return (
    <div style={{ display: "flex", alignItems: "center", gap: "0.5rem", color: "white" }}>
      <span>Halo, <strong>{member.name}</strong> ({member.tier})</span>
      <button onClick={logout}>Logout</button>
    </div>
  );
}
```

Tambahkan ke `app/components/Navbar.tsx` (sudah `"use client"` karena pakai `usePathname`):

```tsx
import { UserBadge } from "./UserBadge";
// ...di dalam <nav>, setelah <ThemeToggle/>:
<div style={{ marginLeft: "auto" }}>
  <UserBadge/>
</div>
```

**Demo ke murid:** buka `/`, `/blog`, `/dashboard` — badge "Halo, Rizki (Gold)" muncul konsisten di
semua halaman tanpa prop-drilling. Klik Logout → cuma Navbar yang re-render, isi halaman blog
(server-rendered) tidak ikut refetch/reload. Buka DevTools → Elements → cari komponen `UserBadge`,
tunjukkan dia satu-satunya bagian yang re-render saat context berubah — ini masuk ke pembahasan
"Re-render Hell" di slide 4-5.

## Step 5 (opsional, kalau waktu cukup) — Demo Re-render Hell

Tantang murid: taruh `console.log("Navbar render")` di `Navbar.tsx` dan
`console.log("UserBadge render")` di `UserBadge.tsx`. Klik Login/Logout berkali-kali, lihat di
console browser — cuma `UserBadge` yang log, `Navbar` (parent) tidak re-render karena dia bukan
consumer `UserContext`. Ini bagus buat tunjukkan `UserContext` disini kecil, jadi belum kena
masalah "satu context raksasa" di slide 4. Tanyakan ke murid: "Kalau context ini isinya juga cart
belanja yang berubah tiap detik, apa yang bakal kejadian?" → jawaban: semua consumer termasuk yang
nggak butuh cart, ikut re-render → saatnya split context atau naik ke Zustand.

## Setelah Live Coding — Quick Check (slide 6)

1. Kenapa Context nggak bisa dipakai di Server Component? → Context bergantung pada React runtime (hooks) yang cuma hidup di Client Component.
2. Provider butuh `'use client'`, apakah children yang dipass ikut jadi Client Component? → Tidak — children yang di-pass sebagai props (seperti di `layout.tsx`) tetap Server Component.
3. Masalah utama Context saat state sering berubah? → Semua consumer re-render tiap value berubah — bisa jadi performance issue di aplikasi besar.

## Tugas Mandiri (Homelab, slide 7)

1. **ThemeContext** — sudah ada contohnya di project ini (`ThemeProvider`), minta murid baca ulang & jelaskan ke teman sebangku kenapa itu valid.
2. **Split Context** — pisahkan `UserContext` yang baru dibuat dari context lain kalau nanti ditambah (misal `CartContext`), verifikasi lewat React DevTools Profiler bahwa update satu context nggak bikin consumer context lain re-render.
3. **Test Re-render** — pakai React DevTools Profiler, record saat Login/Logout, hitung komponen yang re-render.
4. **Context vs Zustand** — buat versi `CartContext` dua kali: satu pakai Context (pola sama seperti `UserContext` ini), satu pakai Zustand. Bandingkan boilerplate & DX. Bawa ke Bab 3 (Zustand: Store Factory + Integrasi Next.js).
